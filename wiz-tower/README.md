# wiz-tower

An adversarial tower-defense game whose waves are generated by a tiny learned adversary
that reads your defense and attacks its gaps ‚Äî all client-side, no backend. See
[`docs/kickoff/`](docs/kickoff) for the full design (`tower-defense-design-plan.md`), the
engine contract (`PHASE0.md`), the interface spec (`lib.rs`), and the Wave DSL grammar
(`wave.ebnf`).

## Status: Phase 0 ‚Äî the deterministic sim (complete)

Phase 0 is the **engine and nothing else** ‚Äî no renderer, no model, no search (those are
the Phase 1+ consumers). It's a deterministic, headless simulation in TypeScript, written
once and shared by the Node trainer and the browser (one V8 engine, one source of truth).

What's implemented, each with tests:

- **Fixed-point (`Fx`, Q22.10) + seeded xorshift PRNG** ‚Äî the determinism substrate.
- **7-element type chart** (`typeMult`): the Sonic‚ÜíEarth‚ÜíZap‚ÜíIce‚ÜíFire wheel + Light‚áÑDark.
- **Grid / occupancy / economy**: build & sell walls and towers with cost/refund; the
  skill-tree economy (attunement escalation, tier gates, expedited starting-element T2).
- **Dual flow fields**: `costWalled` (the route mobs take) and `costOpen` (for choosing a
  breach target), BFS-recomputed only on maze change, with deterministic gradient ties.
- **Entities & the tick**: flow-field movement (fliers fly straight, blocked mobs breach),
  leaks, deaths‚Üíbounty, mender regen; towers acquire (priority then stable id order) and
  fire with typing √ó shield √ó splash √ó slow. All in the exact `PHASE0.md ¬ß2` tick order.
- **Wave DSL**: `parseWave` (grammar + semantic validation) and `PlanAttacker` executing a
  wave's telegraphed opener and resolving hidden-reserve guards at decision points.
- **Golden-replay determinism**: the DoD gate ‚Äî a fixed (seed, layout, decision stream)
  reproduces byte-identical `Metrics` on every run.

### Run it

```bash
npm install
npm test            # 61 tests: type chart, fields, combat, parser, determinism
npm run headless    # plays a scripted wave vs a scripted defense, twice ‚Üí identical scorecard
npm run typecheck   # tsc --noEmit
npm run dev         # serves web/ ‚Äî the same sim running in the browser (no renderer yet)
npm run build       # bundles the browser consumer (~9 KB gzipped)
```

## Architecture: one engine, two consumers

Everything routes through the `Attacker` interface (`src/wave.ts`) and the driver loop
(`src/driver.ts`). The sim (`src/sim.ts`) never knows whether a parsed-DSL `PlanAttacker`
(live play / replay) or a branching `SearchAttacker` (Phase 1.5 training, not yet built)
is behind it. Keep game rules in the sim, once ‚Äî if a rule lives in two places it drifts.
