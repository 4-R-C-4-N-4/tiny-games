# wiz-tower

An adversarial, mobile-first tower-defense game whose waves are generated by a tiny learned
adversary that reads your defense and attacks its gaps — **all client-side, no backend, no
API key**. The "AI" is a ~2.5k-parameter net distilled from search over the game's own
deterministic simulator, run by a ~15-line JS forward pass. See [`docs/kickoff/`](docs/kickoff)
for the full design (`tower-defense-design-plan.md`), the engine contract (`PHASE0.md`), the
interface spec (`lib.rs`), and the Wave DSL grammar (`wave.ebnf`).

## Play it

```bash
npm install
npm run dev        # play in the browser
```

Pick a **starting element**, **difficulty** (1–5), **foe** (Search or Net), and attacker
**style**. Build towers and walls on the grid, **Scout** to see the telegraphed opener,
counter-build against it, then **Start Wave**. During a wave you get a couple of tactical
taps (⚡ overcharge, 👁 reveal, 🧱 reinforce). The hidden reserve commits mid-wave where
you're thin; the post-wave recap shows the feint. Economy and Core HP carry across waves.

## Status — Phases 0–4 complete

| Phase | What | Highlights |
|---|---|---|
| **0** | Deterministic headless sim | fixed-point Q22.10, flow-field pathing + breaching, full tick order, Wave DSL parser, golden-replay determinism |
| **1** | First playable (search, no ML) | `SearchAttacker` forks the sim to simulate candidate openers; Canvas renderer + touch controls |
| **1.5** | Reactive reserve (L2) | hidden reserve committed by live branching search at in-wave decision points — real feints/tempo |
| **2** | Distilled model | sim-as-teacher leak surface → 10→64→28 MLP (`weights.json`, 23 KB) → JS forward pass `ModelAttacker` |
| **3** | Close the loop | in-wave verbs, post-wave recap, personality presets, difficulty, one DAgger round |
| **4** | Ship | static build, relative base, offline service worker, GitHub Pages workflow |

**Distilled model quality:** top-1 agreement with the search teacher **45%** vs a 19%
defense-blind baseline (random 3.6%), **4.5% leak regret** when it disagrees. The action
space is the elemental matchup (the "read the defense, hit the weak type" thesis); the
search/L2 opponent still fields fliers/shades/breakers.

## One engine, two consumers

Everything routes through the `Attacker` interface (`src/wave.ts`) and the driver loop
(`src/driver.ts`). The sim (`src/sim.ts`) never knows whether a `SearchAttacker` (live
branching search) or a `ModelAttacker` (distilled net) is behind it — the same interface the
training search and the shipped model both implement. Game rules live in the sim, once.

## Commands

```bash
npm run dev         # play in the browser
npm test            # 83 tests: sim, fields, combat, parser, determinism, search, model, game, verbs
npm run typecheck   # tsc --noEmit
npm run headless    # scripted wave vs scripted defense, twice → identical scorecard
npm run build       # static build to dist/ (~24 KB gzip incl. weights + service worker)
npm run preview     # serve dist/ on :5199
npm run verify:web  # headless Chromium end-to-end check (needs `npm run preview` running)
npm run calibrate   # probe-strength sweep for the distillation teacher
npm run train       # distill weights.json (a DAgger round if a prior model exists)
```

## Deploy

`git push` to `main` runs `.github/workflows/deploy.yml`: build → GitHub Pages. The app is a
static folder (relative asset paths) with an offline service worker, so it runs from any path
and works offline after first load. No server, no headers, no WASM.

## What's next (stretch)

- **Phase 5 — L3 cross-wave strategist.** Give the attacker memory + a multi-wave plan so it
  models your build tendencies and sets traps across waves (over-defend one side for three
  waves, punish the next three). This is where "feels like a mind" fully lives.
- **Tuning.** The `config.ts` cost/stat tables are deliberately placeholder — balance the
  breach/maze curve, difficulty pacing, and verb economy against real play.
